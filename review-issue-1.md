## Should-fix
- [ ] `apps/frontend/src/lib/components/TaskContent.svelte:30-32` — The project pill stops `pointerdown` and `pointerup` propagation but **not `click`**. The row's tap-to-edit path runs from `onUp` (PointerEvent), so the row itself is now correctly shielded — but the place/link/project pills inside the same `<span>` all show identical defensive triplets (`onclick + onpointerdown + onpointerup` stopPropagation) only because of legacy fear. The project pill's `filterByProject` already calls `e.stopPropagation()` on the click. The `onpointerup` addition is fine; consider extracting a `swallowPointer` action so all four pill kinds use one source of truth — three independent copies of the same triple-stop is the kind of thing the next regression will hide in.
- [ ] `apps/frontend/src/lib/components/TaskItem.svelte:31-32` (pill onpointerup stopPropagation) — when the user presses on the row body (`onDown` runs, sets `active=true`, `hadDown=true`) then drags onto a pill and releases there, the row's `onUp` never fires. `active` and `hadDown` remain `true`, `lock` may be stuck at `"swipe"`, `tx` stays non-zero, and `swipeOpen.id` is not cleared. The state self-heals on the next `onDown`, but during the gap any reactive `$effect` that reads `tx`/`lock` (e.g. the cross-row close effect at line 400) sees stale state and the visible tray stays open. Worth either letting `pointerup` bubble (the row's `hadDown` gate now makes that safe — the row would just no-op a tap because `moved` is false relative to its own stored startX) or explicitly cleaning state in a row-level `pointerup` capture-phase listener.
- [ ] `apps/frontend/src/lib/components/TaskItem.svelte:344-353` (onCancel) — old behaviour `settle()` was changed to `closeTray()`, which is the right *safety* call, but the comment claims "system-cancelled gesture … must never auto-fire" — a real user finishing a fast horizontal flick can also produce `pointercancel` (e.g. the touch crosses into a captured region, or iOS interrupts for the system gesture). Now even a deliberate, completed swipe past the trigger threshold that ends in cancel will silently swallow the action. That's defensible, but the comment is misleading and there's no logging/haptic to tell the user "we cancelled your action". Worth at least a `tapLight()` so the dropped gesture isn't silent.
- [ ] No tests. The issue's acceptance criteria are concrete behaviours ("a single tap performs exactly one action", "tapping a pill … does not also activate the parent task row", "no accidental edit-mode activation during scroll/drag"). Each one is testable with Playwright touch emulation or a unit test that drives synthetic PointerEvents at the gesture state machine. For a fix whose entire purpose is preventing a regression class, shipping zero tests is the largest gap in this PR.
- [ ] Issue scope explicitly lists "section headers" in the audit; `DailySection.svelte:54-66` was not touched. Section header swipe uses raw `e.clientX` deltas without a `touch-action` lock other than `pan-y`, and `TaskItem.onDown` stops propagation, so the existing pill-stopPropagation also prevents the section-header swipe from initiating when the touch starts on a pill. None of this is verified one way or the other in the PR — the candidate's commit message implies a full audit but the diff only touches three files.

## Nits
- `apps/frontend/src/lib/components/FilterBar.svelte:336-343` and `TaskItem.svelte:632-638` — backdrop now `stopPropagation`s `pointerdown`/`pointerup` and closes on `click`. Since the backdrop is portalled to `<body>`, ancestor-bubbling `stopPropagation` is doing nothing meaningful; the load-bearing change is moving the close from `onpointerdown` to `onclick`. The `stopPropagation` calls are cargo. Replacing them with a single comment explaining the down→click delay would be clearer.
- `TaskItem.svelte:142` — comment says "stale coordinates", but the actual failure mode is `moved === false` (because `lastX === startX` from a prior gesture), which routes into `startEdit()`. The fix is right; the comment is one level off from the real bug.
- `TaskItem.svelte:317-320` — `hadDown` is captured into `wasDown` then immediately cleared. Fine, but the `else if (lock === null && wasDown)` branch is the only consumer; the other branch (`lock === "swipe"`) doesn't gate on `hadDown`. That's correct (a swipe lock implies `onMove` ran which implies `active`), but a comment to that effect would prevent a future "harden everything" PR from breaking it.
